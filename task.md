Проект модуля 7. Mini-Launchpad — от учебных модулей к готовому WEB3-продукту
Поздравляем с выходом в финальный этап модуля! Вы изучили ключевые концепции Solana и теперь готовы собрать из них работающий продукт. В этой проектной работе вам предстоит починить, развернуть и запустить Mini-Launchpad — систему для выпуска токенов с комиссией в долларах, использующую оракул цен и off-chain сервис.

Вы прошли путь от базовых концепций до полноценного контура смарт-контрактов:
Оракулы — научились создавать PDA с состоянием, обновлять данные с проверкой авторизации и защищаться от устаревших цен (staleness).
Token Factory — освоили создание Mint и ATA через CPI-вызовы, безопасную математику для расчёта динамических комиссий.
Безопасность и оптимизация — научились экономить Compute Units, писать негативные тесты и проводить аудит кода.
Также вы вынесли интеграцию за пределы блокчейна:
Backend-feeder — написали сервис, который получает данные из внешних API и отправляет их в сеть.
Наблюдаемость — настроили WebSocket-слушатель для отслеживания on-chain событий.
Теперь пришло время собрать все кирпичики воедино.
Что вы будете собирать
Mini-Launchpad состоит из трёх компонентов:
Оракул (sol_usd_oracle) — хранит актуальную цену SOL/USD.
Launchpad-контракт (token_minter) — выпускает токены с фиксированной комиссией в USD.
Backend-сервис — обновляет цену в оракуле и отслеживает события создания токенов.
В некоторых случаях полезно настроить туннель, например ngrok), чтобы кошелёк мог обращаться к вашему локальному валидатору, когда приложение кошелька ходит в RPC с удалённого сервера, адрес localhost для него недоступен. Подробнее здесь — Localhost RPC Endpoints (Backpack).
Ваша задача — сначала починить, затем развернуть.
Задание
Скачайте начальный код проекта здесь.
В репозитории вас ждёт заготовка проекта с намеренно оставленными пробелами. В нескольких местах оставлены todo!(), а часть тестов специально испорчена и помечена как TODO(student). Часть проверок на старте падает специально.
Свой контур приложения. Часть задания — развернуться под своими ключами со своими адресами программ (смарт-контрактов). Нужно сгенерировать свои keypair для оракула и для launchpad, подставить их в program_id в коде (например, declare_id! в Anchor) и в конфигурации (Anchor.toml), после чего собирать и деплоить уже с этими идентификаторами. У каждого студента должен получиться свой экземпляр Mini-Launchpad со своими адресами в сети.
Сценарий работы разбит на две фазы:
Сначала вы чините учебные папки program/ и backend/.
Затем переносите логику в свой контур деплоя и поднимаете полноценное приложение под своими program_id.
Шаг 0. Подготовка репозитория
Создайте публичный репозиторий на GitHub для проекта Mini-Launchpad.
Загрузите исходный код проекта (первый коммит) — это будет базовая версия с todo()-комментариями.
Все последующие изменения оформляйте отдельными коммитами — это поможет ревьюеру отслеживать ваш прогресс.
Шаг 1. Починить program
Перейдите в program/ и найдите места, помеченные как todo!() или TODO(student).
Вам нужно: 
восстановить логику обновления цены в sol_usd_oracle;
восстановить расчёт комиссии минта в token_minter;
исправить намеренно неверные ожидания в LiteSVM-тестах.
Проверка: bashcd program yarn installanchor build yarn run ts-mocha -p ./tsconfig.json -t 1000000 "tests/**/*.ts"
Шаг 2. Починить backend-task
Перейдите в backend-task/ и восстановите helper-функцию преобразования цены в fixed-point формат. Также проверьте unit-тесты: один из них намеренно сломан и должен быть исправлен.
Проверка:bashcd backend-taskcargo test
Шаг 3. Подготовить свой контур деплоя
После того как учебные задания пройдены, переходите к своей версии проекта. Перед деплоем подставьте в program_id свои сгенерированные ключи программ (оракул и token_minter) — так вы задеплоите контракты под своими адресами. Поднимите локальный валидатор (solana-test-validator), соберите программы (anchor build) и разверните их в локальной сети.
Чтобы не запутаться, после генерации своих адресов пройдитесь по проекту в таком порядке:
On-chain часть program/: подставьте свои program_id в declare_id! у обоих контрактов и в program/Anchor.toml.
Frontend frontend/: откройте frontend/app/config.ts и замените константы ORACLE_PROGRAM_ID и MINTER_PROGRAM_ID на свои адреса.
Для удобства в папках program/scripts/ и program/migrations/ есть скрипты (например, init-local.js или deploy.ts), которые помогут инициализировать стейт оракула и конфигурацию минтера. Обязательно запишите полученные адреса PDA.
Если вы используете одни и те же ключи программ для localnet и devnet, то во фронтенде достаточно один раз обновить frontend/app/config.ts и дальше просто переключать сеть в интерфейсе.
Кошелёк для локальной разработки. Для тестирования с локальным RPC (http://localhost:8899 или http://127.0.0.1:8899) рекомендуется использовать Backpack — он корректно работает с локальным валидатором. Многие другие кошельки (в том числе Phantom) обращаются к RPC с удалённых серверов, из‑за чего адрес localhost для них указывает «на себя» сервера, а не на ваш компьютер и подключение к вашему ноду не работает. Подробнее про ограничения localhost RPC и обход через туннель (ngrok и другие) — в документации Backpack.
Шаг 4. Настройка и запуск Backend-сервиса
Перейдите в папку backend/. Скопируйте .env.example в .env и заполните недостающие данные:
Публичные ключи развёрнутых программ (ORACLE_PROGRAM_ID, MINTER_PROGRAM_ID).
Адрес PDA-аккаунта состояния оракула (ORACLE_STATE_PUBKEY).
Путь к вашему локальному кошельку (BACKEND_KEYPAIR_PATH).
После этого запустите сервис командой cargo run. Если всё сделано верно, вы увидите, как backend начинает регулярно пушить новую цену в блокчейн.
Шаг 5. Деплой в Devnet
После деплоя и запуска backend поднимите фронтенд и проверьте полный сценарий:
оракул получает цену;
backend отправляет обновления в сеть;
frontend подключается к localnet;
через интерфейс можно сминтить токен;
backend видит и логирует событие TokenCreated.
Конечная цель — развернуть готовую систему в Devnet и продемонстрировать её работу:
Убедитесь, что обе программы (оракул и launchpad) задеплоены в Devnet.
Backend-сервис должен стабильно обновлять цену через оракул и отслеживать события в Devnet.
Обновите README.md в вашем репозитории, добавив:
адреса контрактов в Devnet;
2-3 ссылки на успешные транзакции создания токенов.
Шаг 6. Деплой в Devnet и сдача результата
Конечная цель — развернуть всё в Devnet и зафиксировать результат. После успешного деплоя подготовьте для сдачи:
Адреса ваших контрактов: program id оракула, program id launchpad (token_minter), адрес PDA состояния оракула.
Ссылки на успешные транзакции: одна или несколько транзакций минта токена через ваше приложение в Devnet (ссылки в Explorer, например: https://explorer.solana.com/tx/...?cluster=devnet).
Так вы покажете, что цепочка «оракул → launchpad → backend» работает не только локально, но и в реальной тестовой сети.
Как быстро запустить проект
В корне репозитория есть Makefile с основными командами:

### Подготовка (один раз)
```bash
make install   # зависимости: program (yarn), frontend (npm) 
Полный цикл на localnet
Валидатор (отдельный терминал, не закрывать):

make validator 
Поднимает solana-test-validator. Убедитесь, что есть keypair и SOL: solana airdrop 10 при необходимости.
Сборка и деплой контрактов:

make build
make deploy 
Инициализация oracle и minter (один раз после деплоя):

make init 
В выводе будет ORACLE_STATE_PUBKEY — скопируйте его в backend/.env (см. backend/.env.example).
Backend (второй терминал):

make backend 
Нужен файл backend/.env с RPC, program ID и ORACLE_STATE_PUBKEY.
Frontend (третий терминал):

make frontend 
Откройте в браузере URL из вывода dev-сервера. В кошельке выберите сеть Localhost / Custom RPC: http://127.0.0.1:8899.
Тесты
Когда всё починено и валидатор не занят (порт 8899 свободен):

make test 
Запускаются LiteSVM-тесты (oracle + minter) без реальной сети.
Итого: make install → в одном терминале make validator → make build → make deploy → make init → в других терминалах make backend и make frontend. Проверка: make test.
Задание можно считать выполненным, если:
в program/ проходят все LiteSVM-тесты;
в backend/ проходят все unit-тесты;
контракты задеплоены под вашими program_id;
backend успешно отправляет обновления цены;
через frontend можно сминтить токен;
у вас есть адреса программ, PDA и ссылка на успешную транзакцию в Devnet.
